## 1-1. 문서 해석 원칙

- `요구사항`은 업무상 반드시 보여야 하는 흐름과 규칙을 뜻한다.
- `오른쪽 Drawer + 내부 탭` 같은 상세 표현 방식은 요구사항 원문이라기보다, IA/화면설계에서 채택한 통합 설계 패턴이다.
- 따라서 `탭/드로어 구조`는 절대 고정 요구사항이 아니라, `요구사항을 더 잘 보여주기 위한 설계 방식`으로 해석한다.
- 이후 추가 조정 시에도 핵심 판단 기준은 `요구사항 반영도`이며, 탭 구조 자체는 재조정 가능하다.

## 2. 공통 반영 사항

### 2-1. 권한/접근 제어

- 역할별 메뉴는 기존 역할 구분을 유지하되, 직접 URL 접근까지 차단하도록 403 화면을 추가했다.
- 계약계열 화면은 `계약담당자`와 `가격등록자/임원`을 별도 시연 권한으로 나눠 화면 상단에서 전환 가능하게 했다.

`고객사 요구사항 강조포인트`
- 메뉴를 숨기는 수준이 아니라, 권한 없는 사용자가 직접 주소를 입력해도 진입하지 못하게 표현했다.
- 낙찰관리 등 핵심 화면에서 `가격등록자/임원`과 `계약담당자`의 차이를 실제 액션 가능 여부로 분리했다.

### 2-2. 상태 체계

- 상태를 하나로 합치지 않고 아래 3개 축으로 분리했다.
  - `contractStatus`
  - `bondStatus`
  - `pmsSyncStatus`
- 화면 표시용 상태 배지는 내부 코드와 한글 상태명을 매핑하도록 공통화했다.

`고객사 요구사항 강조포인트`
- `계약확정`과 `PMS 전송완료`를 같은 상태로 처리하지 않았다.
- `보증반려`는 계약상태가 아니라 보증상태로 분리해 표현했다.

### 2-3. 상태 가이드 도움말

- `SCR-S-05`, `SCR-S-07`, `SCR-S-10`, `SCR-S-11`, `SCR-S-12`, `SCR-S-13`에 상태 가이드 버튼을 추가했다.
- 상태 가이드에는 `상태 의미`, `처리 주체`, `가능 액션`, `다음 단계`, `제한 사항`을 넣었다.

`고객사 요구사항 강조포인트`
- 프로토타입 검토 시 상태 해석이 달라지지 않도록, 실제 화면에서 쓰는 상태명과 같은 용어로 설명을 제공했다.

## 4. 의도적으로 단순화한 부분

- 실제 백엔드 연동은 구현하지 않고 시연용 상태 전환과 안내 중심으로 표현했다.
- 사소한 편의 기능, 미세한 검색 조건, 일부 폼 후처리, 완전한 첨부파일 검증은 후순위로 두었다.
- 핵심은 `권한`, `상태`, `프로세스 흐름`, `액션 가능 여부`가 요구사항대로 보이게 하는 데 두었다.

## 4-1. 추가 반영 대기 항목

아래 항목은 사용자 피드백 기준으로 추가 검토/반영 대기 상태다.

### B. 버튼 정책 재검토

- 일부 화면에서 비활성화 처리한 버튼을 다시 검토
- 요구사항/화면설계서 기준으로 아래 3가지 중 어느 방식이 맞는지 판별 필요
  - 버튼 비노출
  - 클릭 시 안내 팝업
  - 비활성 유지
- 우선 검토 대상
  - `SCR-S-05`
  - `SCR-S-07`
  - `SCR-S-10`
  - `SCR-S-11`
  - `SCR-S-12`

### C. 탭/드로어 구조 재검토

- 일부 화면은 탭 수에 비해 데이터 양이 적어, 프로세스 이해보다 UI 분할이 먼저 보이는 문제가 있음
- 기준
  - 탭을 유지할 만큼 정보량이 충분한가
  - 요구사항 설명에 실제 도움이 되는가
  - 한 화면 섹션으로 합치는 편이 더 직관적인가

### D. 변경 근거 문서 보강

- 향후 각 `고객사 요구사항 강조포인트` 아래에
  - 요구사항서 코드
  - 화면설계서 코드
  를 같이 적어 바로 역추적 가능하도록 유지
