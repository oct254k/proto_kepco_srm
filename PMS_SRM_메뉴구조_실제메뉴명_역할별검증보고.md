# PMS/SRM 메뉴 구조 실제 메뉴명 기준 정리 및 역할별 검증 보고

## 1. 목적
- 현재 프로토타입의 실제 메뉴 구조를 코드 기준으로 재정리한다.
- 역할별 LNB 메뉴명과 상세 페이지명을 분리해 정리한다.
- 역할별 접근 제어가 메뉴 노출과 직접 URL 접근 모두에서 일관적인지 점검한다.

## 2. 검토 기준
- 메뉴 기준 소스: `src/lib/menu.ts`
- 역할/기본 권한 기준: `src/lib/types.ts`, `src/lib/role.ts`
- 직접 URL 접근 제어: `src/lib/access.ts`, `src/components/LayoutShell.tsx`
- 실제 라우트 범위: `src/app/**/page.tsx`
- 관리자 권한 참조 UI 정합성: `src/app/a/users/page.tsx`

## 4. 정리 원칙
- 메뉴는 라우트 전환 단위로 본다.
- 탭은 같은 페이지 안의 뷰 전환으로 본다.
- 따라서 `마이페이지 > 기업정보`, `사용자관리 > 내부 사용자/협력업체 관리/권한그룹`처럼 탭은 메뉴가 아니라 화면 내부 구성으로 분리한다.
- 계약담당자의 `가격등록자/임원`은 별도 역할(role)이 아니라 `ContractAuthority` 전환값으로 유지한다.

## 5. 검증 결과

### 5-1. PASS
1. `Sidebar`는 `MENUS[role]`만 렌더링하므로 역할별 메뉴 노출이 코드상 분리되어 있다.
2. `LayoutShell`은 현재 역할과 경로 prefix를 비교해 권한 없는 경로를 `403 접근 제한`으로 차단한다.
3. 역할별 라우트 prefix가 `/b/`, `/v/`, `/c/`, `/a/`로 분리되어 있어 직접 URL 접근 검증 기준이 명확하다.
4. `src/app/**/page.tsx` 기준 실제 업무 라우트는 24개이며, `/login/`을 제외한 모든 업무 라우트가 `src/lib/menu.ts` 메뉴 경로와 1:1로 매핑된다.
5. 코드 검토 기준으로 관리자 역할은 별도 대시보드가 없고, 기본 진입 화면은 `src/lib/access.ts`의 `/a/system/`이다.

### 5-2. 이번 반영 내용
- 로그인 후 진입 경로를 `/{role}/dashboard/` 하드코딩에서 `getDefaultPath(role)` 기반으로 변경했다.
- 이 변경으로 관리자 로그인 시 존재하지 않는 `/a/dashboard/`로 가던 문제를 제거했다.
- 헤더 로고 클릭 경로도 `getDefaultPath(role)` 기준으로 맞춰, 역할 전환/로그인/복귀 경로가 동일한 기준을 쓰도록 정리했다.

### 5-3. 확인 메모
- 일부 페이지 헤더는 메뉴명보다 설명이 길다. 예: `입찰계획·공고 관리 (조회 전용)`, `발주계획 관리 (계약담당자)`.
- 보고 기준은 헤더 문구가 아니라 실제 LNB/상세 메뉴명(`src/lib/menu.ts`)이다.
- 관리자 기본 진입경로(`src/lib/access.ts`의 `/a/system/`)와 관리자 첫 LNB(`사용자관리`)는 여전히 다르다.
- 이번 작업에서는 로그인/헤더 복귀 경로를 기본 진입경로 기준으로 통일했고, **LNB 첫 메뉴 자체를 바꾸지는 않았다.**
- 따라서 관리자의 “첫 진입 화면”과 “첫 번째 LNB 항목”이 다른 구조가 의도인지 추후 IA 기준으로 한 번 더 확인하는 것이 좋다.

## 6. 결론
- 현재 프로토타입의 실제 메뉴 기준은 `src/lib/menu.ts`로 통일하는 것이 맞다.
- 역할 검증 기준도 `role + route prefix` 조합으로 정리 가능하다.
- 후속 작업이 필요하면 이 보고서를 기준으로 IA, 사용자 가이드, 관리자 권한 안내 문구를 동일 용어로 맞추면 된다.
- 특히 관리자 기본 진입 규칙은 `src/lib/access.ts`를 기준으로 보고, 문서도 같은 기준으로 정렬하는 편이 안전하다.
